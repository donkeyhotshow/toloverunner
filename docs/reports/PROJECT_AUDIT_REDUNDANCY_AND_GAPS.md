## 1. Избыточное и мусорное

### 1.1 Дублирование скриптов быстрого запуска

| Файл | Проблема |
|------|----------|
| `quick-start.bat` | Логика установки (pnpm → yarn → bun → npm) и запуска дублируется. Версия в тексте **2.2.0**, в package.json — **2.4.0**. |
| `quick-start.sh` | То же поведение для Linux/Mac. Версия **2.2.0**. |
| `install-fast.ps1` | То же для PowerShell. Версия **2.2.0**. |
| `start-game.bat` | Упрощённый лаунчер (только pnpm/npm, без yarn/bun). Версия **2.2.0**. |

**Рекомендация:** Оставить один способ запуска в README (`npm run dev` / `pnpm dev`) и один скрипт-обёртку при необходимости (например, только `quick-start.sh` + `quick-start.bat` с вызовом `npm run dev` после проверки `node_modules`). Версию брать из `package.json` или не хардкодить. Лишние файлы (`start-game.bat`, при желании — один из quick-start.*) удалить или пометить deprecated.

### 1.2 Пустая или неочевидная структура

| Элемент | Проблема |
|---------|----------|
| Папка `domain/` | Пустая (в проекте нет файлов в `domain/**`). В ARCHITECTURE упоминается DDD, но папка не используется. |
| Папка `types/` | Содержит только `global.d.ts` (расширения `Window` и `Performance`). Корневой `types.ts` тоже расширяет `Window`. Два места для глобальных типов — риск рассинхрона. |

**Рекомендация:** Либо удалить `domain/`, либо завести в ней доменные типы/константы и описать в ARCHITECTURE. Расширения `Window`/глобальные типы свести в один файл (например, оставить только `types/global.d.ts` и импортировать его из `tsconfig` include, а из `types.ts` убрать `declare global` для Window) или явно разделить: `types.ts` — только экспортируемые enum/типы, `types/global.d.ts` — только глобальные augmentation.

### 1.3 Конфигурация и окружение

| Элемент | Проблема |
|---------|----------|
| `.env` | Содержит реальные ключи Firebase (VITE_FIREBASE_*). Файл в `.gitignore`, но наличие в корне — риск утечки при сбросе игнора или копировании репо. |
| `.env.example` | Не описывает переменные Firebase — у разработчиков нет шаблона для локальной настройки Firebase. |
| `vercel.json` → `functions` | Указано `"app/api/**/*.js"`. Папки `app/api/` в проекте нет — конфигурация мёртвая. |

**Рекомендация:** В `.env.example` добавить закомментированные placeholder'ы для `VITE_FIREBASE_*` (без значений). В `vercel.json` удалить секцию `functions` или добавить `app/api`, если бэкенд на Vercel планируется.

### 1.4 Docker и сервер

| Элемент | Проблема |
|---------|----------|
| `Dockerfile.server` | Запуск через `npx ts-node server/index.ts`. В корневом `package.json` нет зависимости `ts-node` — образ при сборке не сможет запустить сервер. |
| Два Dockerfile + docker-compose | Если game-server в проде не используется, можно оставить один Dockerfile для фронта и упростить docker-compose. |

**Рекомендация:** Либо добавить `ts-node` в devDependencies и использовать его в Dockerfile.server, либо собирать сервер в JS (`tsc` в образе) и запускать `node server/index.js`. При одном фронтовом деплое — оставить в репо только один Dockerfile (фронт) и при необходимости второй файл для сервера держать отдельно/документировать.

### 1.5 Документация и версии

| Место | Проблема |
|-------|----------|
| `README.md` | Указан порт **localhost:5173**; в `vite.config.ts` задан `server.port: 3000` — расхождение. |
| `index.html` | Open Graph / Twitter: версия **v2.2.0**; в package.json — **2.4.0**. |
| Лаунчеры (см. 1.1) | Везде зашита версия **2.2.0**. |

**Рекомендация:** Привести порт в README к 3000 (или к фактическому из vite). Версию в index.html и скриптах брать из package.json при сборке или обновить вручную до 2.4.0. Имеет смысл завести один источник версии (например, `import.meta.env.VITE_APP_VERSION` из vite define) и использовать в превью/шаринге.

## 2. Чего не хватает

### 2.1 Окружение и конфигурация

- **Единый источник версии:** В README, QUICK_START, index.html, батниках — разброс 2.2.0 / 2.4.0. Нужен один источник (package.json или env) и ссылка на него в документации.
- **Описание порта:** В README явно указать, что dev-сервер по умолчанию на порту **3000** (или актуальном из vite).
- **.env.example:** Добавить закомментированные переменные Firebase (`VITE_FIREBASE_API_KEY=`, и т.д.) и кратко описать, когда они нужны.

### 2.2 Сборка и деплой

- **Запуск game-server в Docker:** Либо добавить `ts-node` (или сборку TS → JS) в процесс сборки образа и документировать в README/Runbook, либо убрать/упростить Dockerfile.server.
- **Vercel:** Либо удалить мёртвую секцию `functions` в `vercel.json`, либо добавить реальный `app/api` и описать использование.

### 2.3 Структура и типы

- **Папка `domain/`:** Либо удалить, либо описать в ARCHITECTURE назначение и положить туда доменные типы/события (чтобы не было пустой папки).
- **Глобальные типы:** Явно зафиксировать в ARCHITECTURE или в комментариях: что живёт в `types.ts` (экспорты), что в `types/global.d.ts` (augmentation), чтобы не дублировать расширения Window в двух файлах.

### 2.4 Операционная документация

- **Runbook деплоя:** Краткое описание: как поднять фронт через Docker, как (при необходимости) поднять game-server, какие переменные окружения обязательны. Ссылка на .env.example.
- **TROUBLESHOOTING:** Уже есть; при изменении порта (3000) обновить упоминания localhost при необходимости.
