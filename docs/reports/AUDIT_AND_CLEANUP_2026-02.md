## 2. Найденные проблемы и рекомендации

### 2.1 Зависимости

- **`az` (^0.2.3)** — не используется в коде (проверены `components/`, `core/`). **Рекомендация:** удалить из `dependencies`. **Сделано в рамках очистки.**

### 2.2 Документация и скрипты

- **SOLUTION.md** — ссылки на `quick-demo.html` и `cdn-version.html` ведут в никуда (файлов нет). **Рекомендация:** убрать или переписать раздел «Быстрая демо-версия» и «CDN-версия» с актуальными способами запуска (см. QUICK_START.md).
- **Дублирование «быстрый старт»:** QUICK_START.md, SOLUTION.md, walkthrough.md, README — частично пересекаются. **Рекомендация:** оставить один основной QUICK_START.md + README, в остальных давать ссылку на них.
- **quick-start.bat / install-fast.ps1** — указан порт **5173**, в `vite.config.ts` — **3000**. **Рекомендация:** везде использовать 3000. **Исправлено в скриптах.**

### 2.3 PWA и Service Worker

- **Два источника Service Worker:**
  - Ручная регистрация в `index.tsx`: `navigator.serviceWorker.register('/sw.js')` — регистрирует статический `public/sw.js`.
  - **vite-plugin-pwa** генерирует свой SW при сборке (Workbox).
- В итоге может быть зарегистрирован только один SW (тот, что первым успешно активируется), но конфигурация раздвоена — возможны конфликты при обновлениях кэша и версий.
- **Рекомендация:** выбрать один подход:
  - **Вариант A:** оставить только Vite PWA — удалить `public/sw.js` и регистрацию в `index.tsx`; всё кэширование через Workbox в vite.config.
  - **Вариант B:** оставить только кастомный `public/sw.js` — отключить генерацию SW в vite-plugin-pwa (если плагин это позволяет) и не регистрировать его SW.

- **Manifest:** в `vite.config.ts` задаётся `manifest` внутри VitePWA, а в `public/manifest.json` — отдельный файл. При сборке Vite PWA может перезаписать/подменить manifest. **Рекомендация:** один источник истины — либо только конфиг в vite, либо только `public/manifest.json` и не дублировать в плагине.

### 2.4 HTML и ресурсы

- **index.html** — есть и `<link rel="stylesheet" href="/index.css" />`, и в `index.tsx` — `import './index.css'`. В Vite CSS из точки входа подхватывается бандлом, корневой `index.css` не отдаётся по пути `/index.css`. **Рекомендация:** убрать `<link href="/index.css" />` из index.html, чтобы не было лишнего запроса и путаницы. **Оставлено на усмотрение:** при текущей сборке основной стиль идёт из бандла, ссылка может давать 404 в dev.

### 2.5 Безопасность

- **.env** — в .gitignore, в коде используются только `import.meta.env.VITE_*` и `import.meta.env.DEV/PROD`. Секреты не захардкожены. ✅
- **vercel.json** — CSP и заголовки заданы. ✅
- **public/_headers** — блок с CSP закомментирован (Netlify). Стоит либо раскомментировать и согласовать с vercel.json, либо явно указать в Runbook, что для Netlify заголовки не применяются.

### 2.6 Код и архитектура

- **App.tsx** — комментарий про MicroPlankton («Moved to Environment.tsx») — информативный, можно оставить.
- **Два HUD:** одновременно рендерятся `<HUD />` и `<EnhancedHUD />`. По смыслу может быть задумано (базовый + расширенный), но стоит убедиться, что нет дублирования одних и тех же элементов. По текущему аудиту — оставляем как есть, только фиксируем.
- **TECHNICAL_AUDIT_REPORT.md** — уже описывает дублирование аудио (DynamicAudioManager + UnifiedAudioManager). Рефакторинг аудио — по плану из того отчёта.

### 2.7 Константы и версии

- **APP_VERSION** в `constants.ts` и **VITE_APP_VERSION** в `.env.example` — совпадают с `package.json` (2.2.0). При релизе нужно обновлять в трёх местах или вынести в один источник (например, только package.json + чтение в билде).
