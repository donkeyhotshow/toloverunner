# Что улучшить и стабилизировать

Приоритет: **стабильность → предсказуемость → функциональность**.

---

## ✅ Уже исправлено

- **store.ts:** подписка на `status` типизирована; в dev на `window` вешается только `getState`, `setState` только при VQA.
- **Player.tsx:** в `useEffect` для событий player-collect/hit/graze добавлены зависимости `[isInvincible, lives, isSpeedBoostActive, isImmortalityActive, activeCharType]`; магические числа вынесены в константу `PLAYER_PHYSICS`.
- **gameSlice:** добавлены ограничения для счёта (0 … MAX_SCORE), жизней (0 … maxLives), скорости (MIN_SPEED … MAX_SPEED); используются `GAMEPLAY_CONFIG.MIN_SPEED`, `MAX_SPEED`, `MAX_SCORE`.
- **EnhancedControls:** при размонтировании вызывается `inputManager.destroy()`; при монтировании `inputManager.init()`; добавлен флаг `_listenersAttached` и идемпотентная установка слушателей.
- **Убраны частичные `any`:** UnifiedAudioManager, StabilityManager, GeometryPool, Player (material type).

## 1. Стабильность

### 1.1 Слушатели событий (EventListener)

Во всех компонентах, где есть `addEventListener`, cleanup в `useEffect` уже есть (return с `removeEventListener`). Рекомендации:

- **Player.tsx:** в аудите отмечены возможные stale closures — зависимости эффекта пустые `[]`. Если обработчики используют `isSpeedBoostActive`, `isInvincible`, `lives` и т.д., добавить их в массив зависимостей, чтобы не было устаревших значений.
- **EnhancedControls:** класс с ручной подпиской — убедиться, что `destroy()` вызывается при размонтировании и снимает все слушатели.

### 1.2 Экспозиция store в `window`

**Сейчас:** в dev и при `VQA_EXPOSE_STORE` в `window.__TOLOVERUNNER_STORE__` попадают `getState` и `setState`.

**Рекомендация (по TECHNICAL_AUDIT_REPORT):**

- В production не вешать store при включённом VQA — только в тестовом/CI окружении с явным флагом.
- Для dev по желанию экспонировать только чтение:  
  `get: () => ({ getState: get })` — без `setState`, чтобы снизить риск случайной порчи состояния из консоли.

### 1.3 Таймеры и RAF

- **store.ts:** аудио-цикл через `requestAnimationFrame` корректно останавливается при смене `status` (подписка на `state.status`).
- В других местах: при любом `setInterval` / `setTimeout` / `requestAnimationFrame` в `useEffect` обязательно возвращать очистку (clearInterval / clearTimeout / cancelAnimationFrame).

## 2. Предсказуемость и качество кода

### 2.1 Magic numbers

Вынести в константы (например, в `constants.ts` или рядом с компонентом):

- **Player.tsx:** значения пружины/демпфинга (springK, damping и т.п.) — оформить как `PLAYER_PHYSICS` или аналогичный объект с комментариями.
- Другие «магические» числа в физике/анимации — дать осмысленные имена и вынести в один слой констант.

### 2.2 Валидация и границы

- **store/gameSlice.ts:** при обновлении счёта, жизней, скорости — ограничивать значения сверху/снизу (min/max), чтобы не было отрицательных или нереалистичных чисел.
- Ввод с клавиатуры/сети (если будет) — проверять типы и диапазоны перед записью в store.

### 2.3 Типы

- По возможности убрать оставшиеся `any` (поиск по проекту: `: any`, `as any`) и заменить на конкретные типы или `unknown` с проверками.
