# ADR 0001: Стратегия стабилизации ToLOVERunner

**Статус:** Принято  
**Дата:** 2026-03-08  
**Контекст:** Унификация качества кода, тестов и документации; снижение технического долга при приоритете стабильности и предсказуемости.

---

## Контекст и проблема

- Необходимо устранить P0/P1 TypeScript- и ESLint-ошибки, начиная с критичных путей (core, store, components).
- Тестовые команды в README и package.json должны совпадать с CI; интеграционные и E2E тесты — обязательный прогон в CI.
- Документация (README, ARCHITECTURE, health-отчёты) должна соответствовать текущему коду и скриптам.
- Требуется единая стратегия стабилизации с возможностью отката и фичефлагов для UX-изменений.

---

## Решение

### 1. TypeScript и компиляция

- **Приоритет:** Исправлять блокеры компиляции (ошибки `tsc --noEmit`) в первую очередь. Модули: `InstancedLevelObjects` и все импорты по цепочке (core, infrastructure, types, constants).
- **Подход:** Строгий режим (strict, noUncheckedIndexedAccess) сохраняем; при необходимости точечно ослаблять только в файлах с обоснованием (комментарий + тикет).
- **Откат:** Временные правки типов — только через `// @ts-expect-error` с номером задачи и сроком снятия.

### 2. ESLint и классификация P0/P1

- **P0:** Ошибки парсера, `no-undef`, небезопасное использование any (`@typescript-eslint/no-unsafe-*`), критические `react-hooks/exhaustive-deps` в горячих путях.
- **P1:** `@typescript-eslint/no-unused-vars`, `@typescript-eslint/no-explicit-any`, `no-empty` (без allowEmptyCatch), `react-refresh/only-export-components`.
- **Снижение долга:** Пошагово по модулям: сначала **core**, затем **store**, затем **components** (World → System → UI). Базовый отчёт и классификация — в `docs/reports/ESLINT_BASELINE.md`.

### 3. Тесты и CI

- **Синхронизация:** Все команды тестирования описаны в README и реализованы в `package.json`: `test`, `test:coverage`, `test:integration`, `test:e2e`, `security:audit`, `build:prod`.
- **Vitest:** Интеграционные тесты (`tests/integration/**`) включены в основной прогон `npm run test` (убраны из exclude в `vitest.config.ts`).
- **Playwright:** E2E запускаются через `npm run test:e2e`; в CI используется `webServer` в `playwright.config.ts`: сборка + `vite preview --port 3000`, чтобы E2E работали без ручного старта сервера.
- **Обязательный прогон в CI:** type-check → lint → security:audit → unit/integration tests → установка браузеров Playwright → E2E. При падении любого шага пайплайн останавливается (кроме отдельного job Security Scan).

### 4. Документация

- **README:** Актуальный список скриптов (dev, build, build:prod, type-check, lint, test, test:coverage, test:e2e, test:integration, security:audit). Упоминание ручного QA через `test-runner.html`.
- **docs/ARCHITECTURE.md и health-отчёты:** Обновлять при изменении структуры проекта и скриптов; версия и дата в начале документов.

### 5. UX-улучшения (HUD / мобильный интерфейс)

- **Подход:** Малые изменения (1–2) с **фичефлагами** в `constants.ts` (`FEATURE_FLAGS`). Каждое изменение — отдельный флаг с возможностью отключения без деплоя кода.
- **Реализованные флаги (2026-03-08):**
  - `HUD_COMPACT_MOBILE` — компактные отступы верхней панели HUD на мобильных (p-2 на малых экранах, p-4 на sm+). Используется в `ComicTopBar`.
  - `MOBILE_TOUCH_HIT_AREA` — явный минимальный размер зоны нажатия 44px для мобильных кнопок (a11y). Используется в `MobileControls`.
- **План отката:** В `constants.ts` выставить нужный флаг в `false`, сохранить и задеплоить. При необходимости — откат коммита и повторный деплой. Документировать в CHANGELOG.

## Альтернативы

| Вариант | Плюсы | Минусы |
|--------|--------|--------|
| Ослабить strict TypeScript | Быстро убрать ошибки компиляции | Рост риска рантайм-ошибок, отход от приоритета стабильности |
| Вынести E2E в отдельный workflow | Меньше время основного CI | Сложнее гарантировать «всё зелёное» перед merge |
| Не включать интеграционные в `test` | Разделение unit/integration | Дублирование вызовов в CI и расхождение с README |

---

## Последствия

- **Плюсы:** Единый источник правды для скриптов и документации; предсказуемый CI; явная классификация ESLint; возможность быстрого отката UX через флаги.
- **Минусы:** Первый проход по ESLint/TS может занять время; E2E в CI требуют сборки и времени на прогон.
- **Мониторинг:** Следить за временем CI и при росте — вынести тяжёлые E2E в отдельный job с условным запуском (например, только на main/develop).
